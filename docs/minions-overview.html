What We Built

A complete agentic managed service platform, from infrastructure to agents. Every component was designed for production operation, observability, and governed autonomy.

Foundation
Orchestrator & Intent Engine
Multi-turn LLM orchestrator that classifies natural language requests into structured intents and builds a directed acyclic graph (DAG) of agent calls. Handles parallel fan-out, sequential pipelines, retry, and dead-letter escalation.
Governance Layer
Toolshed MCP Server (Rust)
A high-performance Rust proxy implementing the Model Context Protocol. Every tool call from every agent passes through per-agent allowlists, token-bucket rate limiting, and structured JSON audit logging before execution.

Choose a trigger type to see which agents activate. Complex requests chain all seven; simple queries dispatch just one or two.

Key Design Decisions
🔀 DAG Dispatch, Not Linear Scripts
The Orchestrator builds a dependency graph, not a hardcoded sequence. Parallel work fans out automatically; sequential stages wait for their input. New agents slot in without rewriting the orchestrator.
📤 Structured Outputs as Inputs
Each agent returns validated JSON. The Code Explorer's findings array feeds directly into Code Writer's instructions. Code Writer's files_changed feeds PR Crafter. The pipeline is data-driven.
🔁 Retry with Dead-Letter Escalation
Failed agents retry up to 3 times with exponential backoff. After that, the failure is dead-lettered — clearly surfaced with a reason and suggestion. Downstream stages are skipped cleanly; no silent corruption.
🆔 Correlation ID Everywhere
A single ID is stamped on every agent call, tool log, session event, and Service Bus message. From the dashboard, you can trace any outcome back to its originating ticket or chat message, across all agents.

How It Works
Two Data Paths — Logs and Database

Observability is served by two complementary stores: a streaming log pipeline for real-time monitoring, and a structured SQLite database for durable session records.

Path 1 — Streaming Logs (Real-Time)
Toolshed MCP Proxy (Rust)
Every tool call from every agent passes through here. It checks the allowlist, applies rate limiting, then writes a structured JSON audit log line to stdout — whether the call succeeded or was blocked.
↓
stdout JSON · one line per tool call
Azure Container Apps Console
Container Apps captures stdout from every container automatically. No log agent to manage — it is built in to the runtime. Bot session_start / session_end events also flow this way.
↓
Container Insights ingestion · ≤2 min lag
Log Analytics Workspace
All logs land in ContainerAppConsoleLogs_CL. KQL queries power every real-time dashboard view. Alert rules (P1/P2/P3) fire from here — no additional pipeline needed.
↓
KQL queries via Azure SDK
Dashboard API & React SPA
Express API on port 3001. Live events pushed to the browser via SSE at /api/live/stream. KQL-backed views for costs, tool stats, and config.
Path 2 — SQLite Database (Durable)
Orchestrator
Writes one row to framework_sessions when a session starts, and one row to minion_runs each time an agent is dispatched. Status, attempt count, and token totals are updated as the session progresses.
↓
sessions & minion runs
and
Toolshed MCP Proxy (Rust)
Also writes every tool call to the tool_calls table and every allowlist block to the allowlist_denials table — providing a relational, indexed audit trail queryable by agent, tool, or correlation ID.
↓
SQL writes · low-latency · indexed
SQLite — /data/goose-sessions.db
Co-located with the Orchestrator container. Four tables: framework_sessions, minion_runs, tool_calls, allowlist_denials. Backed up to Azure Blob Storage every 15 minutes. RPO: 15 minutes.

SQLite Schema
Four Tables — Session State and Security Audit

The schema is defined in scripts/init-db.sql and applied at container startup. Every table is indexed on correlation ID and agent for fast dashboard queries.

framework_sessions
One row per orchestration session.
session_id  ·  correlation_id
status — active | completed | dead_lettered
minion_count  ·  tool_call_count
created_at  ·  completed_at
minion_runs
One row per agent dispatch.
id — child correlation ID
agent — e.g. code-writer
status — running | completed | failed | dead_lettered
attempts  ·  max_turns  ·  last_error
tool_calls
One row per tool call (all results).
correlation_id  ·  agent
tool_name  ·  params (JSON)
result — success | failure | blocked
duration_ms  ·  output_size_bytes
allowlist_denials
Security audit — blocked calls only.
correlation_id  ·  agent
tool_name — what was attempted
params — what was passed
reason — allowlist_denied | rate_limited
Toolshed Audit Log Format

Every Toolshed log line is structured JSON written to both stdout (Log Analytics path) and the tool_calls table (SQLite path). The correlation_id links it to the originating session in both stores.

// Successful tool call { "ts": "2026-06-17T09:41:22.140Z", "correlation_id": "corr_a1b2c3d4", "minion_type": "code-writer", "tool_name": "filesystem.write_file", "result": "success", "duration_ms": 180 } // Blocked call — written to allowlist_denials table { "ts": "2026-06-17T09:41:25.001Z", "correlation_id": "corr_a1b2c3d4", "minion_type": "code-explorer", "tool_name": "filesystem.write_file", "result": "blocked", "reason": "allowlist_denied" }

Blocked calls are never silently dropped. Every denial lands in allowlist_denials and Log Analytics — two queryable, alertable records of every governance event.

Proactive Alerting
Know Before Your Users Do

KQL alert rules on Log Analytics fire automatically to PagerDuty or Teams. Every alert tier maps to a specific response time and audience.

P1 — 15 min response
Service Down
  • Orchestrator: no heartbeat for 5 min
  • 5xx error rate >20% sustained 5 min
  • AI Foundry quota exhausted
P2 — 30 min response
Degraded Service
  • Agent failure rate >10% for 15 min
  • Timeout rate >5% for 15 min
  • Service Bus dead-letter >0
  • Container restart loop (3+ in 10 min)
P3 — 2 hr response
Anomalies & Governance
  • Any allowlist denial detected
  • P95 latency >10s for 15 min
  • Container CPU >80% for 10 min
  • Daily spend >2× rolling average
Governance alert: zero allowlist denials should be the normal operating state. A denial means an agent attempted a tool outside its permitted scope — this is a governance event, not just a failure. P3 alerting ensures every denial is reviewed, even when it causes no outage. The allowlist itself is Rust-compiled and version-controlled: changing what an agent can do requires a code change, a PR, and a deployment.

Key architecture principle: Once the Orchestrator dispatches an agent, that agent communicates directly with the Toolshed to call its tools. The Orchestrator only receives the final structured JSON result when the agent completes — it is never in the tool-call path. This means agents run independently and in parallel, each with their own Toolshed connection and per-agent allowlist enforcement.

A "fix this ticket and raise a PR" request triggers the full 6-stage pipeline. Each stage's output feeds directly into the next.

Final security scan before merge
Parallel execution where possible. Single-purpose requests (e.g. "Review PR 342") dispatch just the Code Reviewer. The full pipeline only runs for complex intents like ticket_fix_pr. The Orchestrator builds a dependency graph (DAG) and spawns agents in the correct order, passing each agent's structured JSON output as the next agent's input. All tool calls — filesystem reads/writes, shell execution, GitHub API, ADO queries — happen inside each agent's direct connection to the Toolshed, not through the Orchestrator.

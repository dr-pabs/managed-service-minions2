## ACP WebSocket Message Format

The authoritative spec is [RFN: Streamable HTTP & WebSocket Transport](https://github.com/agentclientprotocol/agent-client-protocol/blob/main/docs/rfds/streamable-http-websocket-transport.mdx) — currently in **Preview** status, implemented in Goose.

---

### Connection Establishment

```
Client                                    Server
  │  GET /acp                               │
  │  Upgrade: websocket                     │
  │  Sec-WebSocket-Key: <key>              │
  │────────────────────────────────────────►│
  │  HTTP 101 Switching Protocols           │
  │  Acp-Connection-Id: <uuid>              │
  │◄────────────────────────────────────────│
  │  ══════ WebSocket Channel ══════════════│
```

After upgrade, all messages are **WebSocket text frames** containing **JSON-RPC 2.0**. Binary frames are ignored. The `Acp-Connection-Id` is delivered in the 101 response header.

---

### Core JSON-RPC Envelope

All messages follow JSON-RPC 2.0:

```
{
  "jsonrpc": "2.0",
  "id": <number | string>,       // Request/Response correlation
  "method": "<method_name>",     // Present on requests only
  "params": { ... },             // Method parameters
  "result": { ... }              // Present on responses only
}
```

### Connection Teardown

When the WebSocket closes (either side), the server cleans up the connection and associated sessions. No special teardown message is required.

| `session/cancel`                  | C→S       | No        | Cancel in-progress turn                    |
| `session/load`                    | C→S       | Yes       | Resume existing session                    |
| `session/close`                   | C→S       | No        | Close a session                            |
| `session/delete`                  | C→S       | No        | Delete a session permanently               |
| `session/list`                    | C→S       | Yes       | List all sessions                          |
| `session/fork`                    | C→S       | Yes       | Fork a session at a message                |
| `request_permission`              | S→C       | Yes       | Server asks client for permission          |
| `session/update`                  | S→C       | No        | Usage update notification                  |
| `notifications/AgentMessageChunk` | S→C       | No        | Streaming agent response chunk             |
| `notifications/AgentThoughtChunk` | S→C       | No        | Streaming thinking/reasoning chunk         |
| `notifications/UserMessageChunk`  | S→C       | No        | Replayed user message                      |

| `notifications/ToolCallUpdate`    | S→C       | No        | Tool result/status update                  |

    - Subscription limits are not exceeded
    - Node.js and npm are installed (for npm-distributed adapters)

## ACP WebSocket Message Format

The authoritative spec is [RFN: Streamable HTTP & WebSocket Transport](https://github.com/agentclientprotocol/agent-client-protocol/blob/main/docs/rfds/streamable-http-websocket-transport.mdx) — currently in **Preview** status, implemented in Goose.

---

### Connection Establishment

```
Client                                    Server
  │  GET /acp                               │
  │  Upgrade: websocket                     │
  │  Sec-WebSocket-Key: <key>              │
  │────────────────────────────────────────►│
  │  HTTP 101 Switching Protocols           │
  │  Acp-Connection-Id: <uuid>              │
  │◄────────────────────────────────────────│
  │  ══════ WebSocket Channel ══════════════│
```

After upgrade, all messages are **WebSocket text frames** containing **JSON-RPC 2.0**. Binary frames are ignored. The `Acp-Connection-Id` is delivered in the 101 response header.

---

### Core JSON-RPC Envelope

All messages follow JSON-RPC 2.0:

```
{
  "jsonrpc": "2.0",
  "id": <number | string>,       // Request/Response correlation
  "method": "<method_name>",     // Present on requests only
  "params": { ... },             // Method parameters
  "result": { ... }              // Present on responses only
}
```

**Client → Server:**
```json

  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "1.0",

}
```

### Connection Teardown

When the WebSocket closes (either side), the server cleans up the connection and associated sessions. No special teardown message is required.

| `session/delete`                  | C→S       | No        | Delete a session permanently               |
| `session/list`                    | C→S       | Yes       | List all sessions                          |
| `session/fork`                    | C→S       | Yes       | Fork a session at a message                |

| `session/update`                  | S→C       | No        | Usage update notification                  |
| `notifications/AgentMessageChunk` | S→C       | No        | Streaming agent response chunk             |
| `notifications/AgentThoughtChunk` | S→C       | No        | Streaming thinking/reasoning chunk         |
| `notifications/UserMessageChunk`  | S→C       | No        | Replayed user message                      |

### Key Differences from MCP

|                | MCP                   | ACP                                                  |
|----------------|-----------------------|------------------------------------------------------|
| HTTP streams   | New SSE per request   | Long-lived GET (1 connection-scoped + 1 per session) |
| `initialize`   | Returns SSE stream    | Returns `200 OK` JSON directly                       |
| WebSocket      | Not defined           | Required client capability                           |
| Identity       | `Mcp-Session-Id` only | `Acp-Connection-Id` + `Acp-Session-Id`               |
| Batch requests | Supported             | Returns 501                                          |
| Cookie support | Optional              | Required                                             |
